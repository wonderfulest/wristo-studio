# Wristo Studio `.wrt` 设计包

## 目标

让创作者可以从 Studio 导出一个完整、可移植的 `.wrt` 文件，并在 Studio 中导入后自动打开为一份新的未保存设计。导入绝不更新或覆盖原有云端设计。

## 范围

- 在编辑器的 Actions 菜单提供“导出 `.wrt`”和“导入 `.wrt`”。
- `.wrt` 包含可直接恢复画布的设计配置与该设计实际引用的素材。
- 导入完成后，画布、属性、图层、预览状态与临时素材恢复到可继续编辑的状态。
- 新设计保留源设计名称并追加“副本”；用户可在应用属性中改名。

不包含服务端上传、共享链接、云端设计记录迁移或向旧版本 Studio 的兼容回写。

## 文件格式

`.wrt` 使用 ZIP 容器，但下载文件名和 MIME 类型按 Wristo 设计包处理。第一版目录结构如下：

```text
<slug>.wrt
├── manifest.json
├── design.json
├── config/config.json
├── elements/
├── assets/
├── fonts/
├── weather/
├── icons/
└── preview.png 或 preview.svg
```

`manifest.json` 是唯一入口，必须声明：

- `format`: 固定值 `wristo-design-package`
- `version`: 当前格式版本 `1`
- `generatedAt`、源设计显示名和画布尺寸
- `design.path`: 设计配置路径，第一版固定为 `design.json`
- 素材清单及其包内路径

设计配置继续使用 Studio 已有的 `RuntimeDesignConfig` 表示；包内元素、图片、字体、AMOLED 天气图标和 AMOLED 普通图标均沿用现有素材包的收集与恢复规则。这样 `.wrt` 能复用已经验证过的资产解析链路，而不是维护两套格式。

## 导出流程

1. 用户在编辑器 Actions 菜单选择“导出 `.wrt`”。
2. Studio 从当前画布生成运行时配置，收集已被元素引用的资源与预览图。
3. Studio 写入 `.wrt` manifest 和内容，浏览器下载 `<slug>.wrt`。
4. 不修改当前设计、云端设计或发布资源。

导出不要求当前设计已保存。对于未保存画布，`designUid` 允许为空，包仍可被导入。

## 导入流程与状态边界

1. 用户在 Actions 菜单选择“导入 `.wrt`”，从本地选择一个文件。
2. 在开始替换画布前，读取 ZIP，并验证扩展名、ZIP 可读性、manifest 的 `format`、`version` 与 `design.path`。
3. 只有全部基础校验通过后才清理现有编辑状态，避免错误文件破坏当前画布。
4. 从 `design.json` 读取配置，恢复包内素材到现有临时素材 store，加载字体和元素，并恢复显示状态、图层顺序、属性和画布缩放。
5. 清空历史记录并以已导入的画布作为新的初始历史快照。
6. 将 `baseStore.id` 和 `designStore.id` 置为空，且移除 URL 中的设计 ID；导入后保存会走现有“创建设计”路径。
7. 将名称设为 `<源名称> Copy`（本地化显示可由 UI 翻译实现），并显示导入成功提示。

## 错误处理

- 非 `.wrt` 文件、不可解压文件、缺少 manifest、格式标识错误、版本不支持、缺少或无效 `design.json`：拒绝导入，保留当前画布，给出可理解的错误提示。
- 包内单个可选素材缺失或损坏：继续加载设计，在提示中列出未能恢复的素材类别；不把外部 URL 当作恢复成功。
- 不支持的未来格式版本：明确提示需要更新 Studio，不能尝试猜测解析。
- 导入过程中发生运行时加载异常：尽力恢复导入前的编辑状态；若不能恢复，刷新页面后仍能从云端 URL 指向的原设计重新加载。

## 测试与验收

- 使用包含背景、图片、字体、AMOLED 天气图标和 AMOLED 普通图标的设计导出 `.wrt`，再导入，画布视觉与元素配置一致。
- 导入后地址栏不包含原设计 ID，且保存会创建新的设计 UID，而不会更新源设计。
- 用空画布导出的 `.wrt` 也能导入并形成可保存草稿。
- 非 `.wrt`、损坏 ZIP、缺少 manifest、未知格式版本和缺少 `design.json` 分别显示错误，且导入前的画布保持不变。
- `npm run build` 通过。

## 兼容策略

`.wrt` v1 只保证由支持 v1 的 Studio 导出和导入。格式演进通过 `manifest.version` 管理：兼容变更增加可选字段；破坏性变更递增版本并在导入端显式分支处理。
